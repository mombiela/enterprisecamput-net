Error Handling with Result
Error Handling with Option ►

Error handling is a crucial aspect of programming, ensuring that your code can gracefully handle unexpected situations. In Rust, the Result type is a powerful tool for managing errors. This topic will guide you from the basics to advanced usage of Result for error handling in Rust.

Introduction to `Result`

The Result type is an enum that represents either success (Ok) or failure (Err). It is defined as:

enum Result<T, E> {
    Ok(T),
    Err(E),
}
  • T: The type of the value returned in the success case.
  • E: The type of the error returned in the failure case.

Basic Usage

Let's start with a simple example:

fn divide(a: f64, b: f64) -> Result<f64, String> {
    if b == 0.0 {
        Err(String::from("Division by zero"))
    } else {
        Ok(a / b)
    }
}

fn main() {
    match divide(4.0, 2.0) {
        Ok(result) => println!("Result: {}", result),
        Err(e) => println!("Error: {}", e),
    }

    match divide(4.0, 0.0) {
        Ok(result) => println!("Result: {}", result),
        Err(e) => println!("Error: {}", e),
    }
}

In this example:

  • The divide function returns a Result<f64, String>.
  • If the division is successful, it returns Ok(result).
  • If there is an error (division by zero), it returns Err(error_message).

Handling `Result` with `match`

The match statement is a common way to handle Result:

fn main() {
    let result = divide(10.0, 2.0);

    match result {
        Ok(value) => println!("Success: {}", value),
        Err(error) => println!("Error: {}", error),
    }
}

Using `unwrap` and `expect`

For quick prototyping, you can use unwrap and expect to handle Result:

  • unwrap: Returns the value inside Ok or panics if it's Err.
  • expect: Similar to unwrap, but allows you to provide a custom panic message.
fn main() {
    let result = divide(10.0, 2.0).unwrap();
    println!("Result: {}", result);

    let result = divide(10.0, 0.0).expect("Failed to divide");
}

Note: Using unwrap and expect is not recommended for production code as they can cause your program to panic.

Advanced Error Handling

Chaining Methods with `and_then` and `or_else`

You can chain multiple operations that return Result using and_then and or_else:

  • and_then: Chains another operation if the result is Ok.
  • or_else: Chains another operation if the result is Err.
fn add_one(x: f64) -> Result<f64, String> {
    Ok(x + 1.0)
}

fn main() {
    let result = divide(10.0, 2.0)
        .and_then(add_one)
        .or_else(|e| Err(format!("Failed: {}", e)));

    match result {
        Ok(value) => println!("Success: {}", value),
        Err(error) => println!("Error: {}", error),
    }
}

Using `?` Operator

The ? operator is a shorthand for propagating errors. It can only be used in functions that return Result.

fn divide_and_add(a: f64, b: f64) -> Result<f64, String> {
    let result = divide(a, b)?;
    add_one(result)
}

fn main() {
    match divide_and_add(10.0, 2.0) {
        Ok(value) => println!("Success: {}", value),
        Err(error) => println!("Error: {}", error),
    }
}

Custom Error Types

For more complex applications, you might want to define custom error types:

#[derive(Debug)]
enum MathError {
    DivisionByZero,
    NegativeLogarithm,
}

fn divide(a: f64, b: f64) -> Result<f64, MathError> {
    if b == 0.0 {
        Err(MathError::DivisionByZero)
    } else {
        Ok(a / b)
    }
}

fn main() {
    match divide(10.0, 0.0) {
        Ok(value) => println!("Success: {}", value),
        Err(error) => println!("Error: {:?}", error),
    }
}

Summary

  • The Result type is essential for error handling in Rust.
  • Use match to handle Result explicitly.
  • unwrap and expect can be used for quick prototyping but are not recommended for production.
  • The ? operator simplifies error propagation.
  • Custom error types provide more flexibility and clarity in error handling.

By mastering Result, you can write robust and error-resistant Rust code.
