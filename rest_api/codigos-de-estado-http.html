Introduction

HTTP status codes are standard responses provided by web servers to indicate the outcome of an HTTP request. These codes are essential for the development and debugging of RESTful APIs, as they inform the client about the success or failure of a request.

Categories of HTTP Status Codes

HTTP status codes are divided into five main categories:

  1. 1xx (Informational): Indicate that the request has been received and the process is continuing.
  2. 2xx (Success): Indicate that the request was received, understood, and accepted successfully.
  3. 3xx (Redirection): Indicate that additional actions are required to complete the request.
  4. 4xx (Client Errors): Indicate that there was an error in the client's request.
  5. 5xx (Server Errors): Indicate that the server failed to fulfill a seemingly valid request.

Common HTTP Status Codes

2xx (Success)

  • 200 OK: The request has succeeded. The response depends on the HTTP method used. 
    HTTP/1.1 200 OK
Content-Type: application/json

{
  "message": "Successful request"
}
  • 201 Created: The request has succeeded and a new resource has been created. 
    HTTP/1.1 201 Created
Content-Type: application/json

{
  "id": 123,
  "message": "Resource created successfully"
}
  • 204 No Content: The request has succeeded, but there is no content to return. 
    HTTP/1.1 204 No Content

3xx (Redirection)

  • 301 Moved Permanently: The requested resource has been permanently moved to a new URL. 
    HTTP/1.1 301 Moved Permanently
Location: https://new-url.com/resource
  • 302 Found: The requested resource is temporarily located at a different URL. 
    HTTP/1.1 302 Found
Location: https://temporary-url.com/resource

4xx (Client Errors)

  • 400 Bad Request: The request could not be understood or was malformed. 
    HTTP/1.1 400 Bad Request
Content-Type: application/json

{
  "error": "Malformed request"
}
  • 401 Unauthorized: The request requires user authentication. 
    HTTP/1.1 401 Unauthorized
Content-Type: application/json

{
  "error": "Authentication required"
}
  • 403 Forbidden: The server understands the request but refuses to authorize it. 
    HTTP/1.1 403 Forbidden
Content-Type: application/json

{
  "error": "Access denied"
}
  • 404 Not Found: The requested resource could not be found. 
    HTTP/1.1 404 Not Found
Content-Type: application/json

{
  "error": "Resource not found"
}

5xx (Server Errors)

  • 500 Internal Server Error: The server encountered an unexpected condition that prevented it from fulfilling the request. 
    HTTP/1.1 500 Internal Server Error
Content-Type: application/json

{
  "error": "Internal server error"
}
  • 502 Bad Gateway: The server, acting as a gateway or proxy, received an invalid response from the upstream server. 
    HTTP/1.1 502 Bad Gateway
Content-Type: application/json

{
  "error": "Bad gateway"
}
  • 503 Service Unavailable: The server is temporarily unavailable, usually due to maintenance or overload. 
    HTTP/1.1 503 Service Unavailable
Content-Type: application/json

{
  "error": "Service unavailable"
}

Practical Examples

Example of a Successful GET Request

GET /api/users/1 HTTP/1.1
Host: example.com

Response:

HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 1,
  "name": "John Doe",
  "email": "john.doe@example.com"
}

Example of a POST Request with Resource Creation

POST /api/users HTTP/1.1
Host: example.com
Content-Type: application/json

{
  "name": "Jane Smith",
  "email": "jane.smith@example.com"
}

Response:

HTTP/1.1 201 Created
Content-Type: application/json

{
  "id": 2,
  "name": "Jane Smith",
  "email": "jane.smith@example.com"
}

Conclusion

Understanding HTTP status codes is fundamental for the development and debugging of RESTful APIs. These codes provide crucial information about the outcome of HTTP requests and help identify and resolve issues efficiently. By familiarizing themselves with the most common status codes, developers can improve communication between the client and the server, ensuring a smoother and more effective user experience.
