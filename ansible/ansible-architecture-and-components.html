Introduction

Ansible is an open-source automation tool used for configuration management, application deployment, and task automation. Understanding its architecture and components is crucial for effectively using Ansible in real-world scenarios. This section will cover the core architecture and key components of Ansible.

Ansible Architecture

Key Concepts

  • Agentless Architecture: Ansible does not require any agent software to be installed on the managed nodes. It uses SSH for communication.
  • Push-based Model: Ansible pushes configurations from a central machine (control node) to the managed nodes.
  • Idempotency: Ansible ensures that applying the same configuration multiple times will not change the system after the first application.

Control Node and Managed Nodes

  • Control Node: The machine where Ansible is installed and from which commands are run.
  • Managed Nodes: The target machines that are managed by Ansible.

Inventory

  • Inventory File: A file that lists the managed nodes. It can be a simple text file or a dynamic inventory script. 
    [webservers]
web1.example.com
web2.example.com

[dbservers]
db1.example.com
db2.example.com

Ansible Components

Playbooks

  • Playbooks: YAML files that define a series of tasks to be executed on managed nodes. 
    - name: Install and start Apache
  hosts: webservers
  tasks:
    - name: Install Apache
      apt:
        name: apache2
        state: present
    - name: Start Apache
      service:
        name: apache2
        state: started

Modules

  • Modules: Reusable, standalone scripts that Ansible runs on your behalf. They can manage system resources, install software, and more.
    • Example: apt, yum, service, copy, template.

Roles

  • Roles: A way to organize playbooks and other files to facilitate reuse and sharing.
    • Directory structure: 
      roles/
  common/
    tasks/
      main.yml
    handlers/
      main.yml
    templates/
    files/
    vars/
      main.yml
    defaults/
      main.yml
    meta/
      main.yml

Plugins

  • Plugins: Extend Ansible's core functionality. Types include:
    • Action Plugins: Modify the behavior of Ansible tasks.
    • Lookup Plugins: Retrieve data from external sources.
    • Filter Plugins: Transform data inside templates.
    • Callback Plugins: Customize output or logging.

Variables and Facts

  • Variables: Used to store values that can be reused throughout playbooks. 
    vars:
  http_port: 80
  • Facts: Automatically gathered information about managed nodes.
    • Example: ansible_facts, ansible_os_family.

Handlers

  • Handlers: Special tasks that are triggered by other tasks. Typically used for service restarts. 
    handlers:
  - name: restart apache
    service:
      name: apache2
      state: restarted

Conclusion

Understanding Ansible's architecture and components is fundamental for leveraging its full potential. The agentless, push-based model simplifies configuration management, while components like playbooks, modules, roles, and plugins provide powerful tools for automation. By mastering these elements, you can efficiently manage and automate your IT infrastructure.
