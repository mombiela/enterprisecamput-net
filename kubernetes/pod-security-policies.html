Pod Security Policies (PSPs) are a critical aspect of Kubernetes security, providing a mechanism to control the security-related aspects of pod creation and updates. This topic will guide you from the basics to advanced concepts of PSPs, ensuring a comprehensive understanding.

Introduction to Pod Security Policies

Pod Security Policies are cluster-level resources that control security-sensitive aspects of pod specification. They allow administrators to define a set of conditions that a pod must meet to be accepted into the system.

Key Concepts

  • Pod Security Policy (PSP): A cluster-level resource that defines a set of conditions that a pod must meet to be accepted into the system.
  • Admission Controller: A piece of code that intercepts requests to the Kubernetes API server prior to persistence of the object, but after the request is authenticated and authorized.
  • Security Context: Defines privilege and access control settings for a Pod or Container.

Creating a Pod Security Policy

To create a PSP, you need to define a YAML configuration file that specifies the security requirements for your pods.

Example PSP YAML

apiVersion: policy/v1beta1
kind: PodSecurityPolicy
metadata:
  name: example-psp
spec:
  privileged: false
  allowPrivilegeEscalation: false
  requiredDropCapabilities:
    - ALL
  volumes:
    - 'configMap'
    - 'emptyDir'
    - 'projected'
    - 'secret'
  hostNetwork: false
  hostIPC: false
  hostPID: false
  runAsUser:
    rule: 'MustRunAsNonRoot'
  seLinux:
    rule: 'RunAsAny'
  supplementalGroups:
    rule: 'MustRunAs'
    ranges:
      - min: 1
        max: 65535
  fsGroup:
    rule: 'MustRunAs'
    ranges:
      - min: 1
        max: 65535

Explanation

  • privileged: Ensures that no privileged containers are allowed.
  • allowPrivilegeEscalation: Prevents privilege escalation.
  • requiredDropCapabilities: Drops all capabilities to minimize the potential attack surface.
  • volumes: Restricts the types of volumes that can be used.
  • hostNetwork, hostIPC, hostPID: Prevents the use of the host's network, IPC, and PID namespaces.
  • runAsUser: Ensures that containers run as a non-root user.
  • seLinux: Allows any SELinux context.
  • supplementalGroups: Requires that containers run with specific supplemental groups.
  • fsGroup: Requires that containers run with specific file system groups.

Applying a Pod Security Policy

To apply a PSP, you need to enable the PodSecurityPolicy admission controller and create appropriate RoleBindings or ClusterRoleBindings.

Enabling the Admission Controller

Edit the Kubernetes API server configuration to include the PodSecurityPolicy admission controller.

--enable-admission-plugins=PodSecurityPolicy

Creating RoleBindings

You need to create RoleBindings or ClusterRoleBindings to bind the PSP to specific users or service accounts.

apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: use-example-psp
  namespace: default
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: Role
  name: use-example-psp
subjects:
- kind: User
  name: example-user
  apiGroup: rbac.authorization.k8s.io

Explanation

  • roleRef: References the role that grants permissions to use the PSP.
  • subjects: Specifies the users or service accounts that the RoleBinding applies to.

Advanced Usage of Pod Security Policies

Combining Multiple PSPs

You can create multiple PSPs to cater to different security requirements for different namespaces or applications.

Example

apiVersion: policy/v1beta1
kind: PodSecurityPolicy
metadata:
  name: restricted-psp
spec:
  privileged: false
  allowPrivilegeEscalation: false
  requiredDropCapabilities:
    - ALL
  runAsUser:
    rule: 'MustRunAsNonRoot'
  seLinux:
    rule: 'RunAsAny'
  supplementalGroups:
    rule: 'MustRunAs'
    ranges:
      - min: 1
        max: 65535
  fsGroup:
    rule: 'MustRunAs'
    ranges:
      - min: 1
        max: 65535
apiVersion: policy/v1beta1
kind: PodSecurityPolicy
metadata:
  name: permissive-psp
spec:
  privileged: true
  allowPrivilegeEscalation: true
  runAsUser:
    rule: 'RunAsAny'
  seLinux:
    rule: 'RunAsAny'
  supplementalGroups:
    rule: 'RunAsAny'
  fsGroup:
    rule: 'RunAsAny'

Explanation

  • restricted-psp: Enforces strict security policies.
  • permissive-psp: Allows more permissive settings for specific use cases.

Conclusion

Pod Security Policies are a powerful tool for enforcing security standards in a Kubernetes cluster. By understanding and implementing PSPs, you can significantly enhance the security posture of your Kubernetes environment. Remember to:

  • Define clear and concise PSPs tailored to your security requirements.
  • Apply PSPs using RoleBindings or ClusterRoleBindings.
  • Regularly review and update PSPs to adapt to evolving security needs.

By mastering PSPs, you ensure that your Kubernetes clusters are secure and compliant with best practices.
