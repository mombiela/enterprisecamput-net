Introduction to ILE

Integrated Language Environment (ILE) is a programming model introduced by IBM for the AS/400 (now IBM i) systems. It allows for the integration of different programming languages and provides a more modular and efficient way to develop applications.

Key Concepts of ILE

  • Modules: The smallest unit of code in ILE, which can be written in different languages (RPG, COBOL, C, etc.). Modules are compiled but not executable on their own.
  • Programs: Executable objects created by binding one or more modules together.
  • Service Programs: Collections of reusable procedures and functions that can be shared among different programs.
  • Binding Directory: A list of modules and service programs that can be used during the binding process.
  • Activation Groups: Logical containers that manage the runtime environment for ILE programs and service programs.

Modules

Modules are the building blocks of ILE. They contain procedures and functions that can be reused across different programs.

Creating a Module

To create a module in RPG, you use the CRTRPGMOD command. Here is a simple example of an RPG module:

**FREE
Dcl-Proc HelloWorld;
  Dcl-S message Char(50);
  message = 'Hello, World!';
  Dsply message;
End-Proc;

In this example:

  • Dcl-Proc declares a procedure named HelloWorld.
  • Dcl-S declares a variable named message.
  • Dsply displays the message on the screen.

Programs

Programs are created by binding one or more modules together. This is done using the CRTPGM command.

Creating a Program

Here is an example of creating a program from a module:

  1. Create the module:

    CRTRPGMOD MODULE(MYLIB/HELLOMOD) SRCFILE(MYLIB/QRPGLESRC) SRCMBR(HELLOMOD)

  2. Create the program:

    CRTPGM PGM(MYLIB/HELLOPGM) MODULE(MYLIB/HELLOMOD)

Service Programs

Service programs are collections of procedures that can be used by multiple programs. They are created using the CRTSRVPGM command.

Creating a Service Program

Here is an example of creating a service program:

  1. Create the module:

    CRTRPGMOD MODULE(MYLIB/MYSRVPGM) SRCFILE(MYLIB/QRPGLESRC) SRCMBR(MYSRVPGM)

  2. Create the service program:

    CRTSRVPGM SRVPGM(MYLIB/MYSRVPGM) MODULE(MYLIB/MYSRVPGM)

Binding Directory

A binding directory is used to simplify the process of binding modules and service programs. It contains a list of modules and service programs that can be referenced during the binding process.

Creating a Binding Directory

Here is an example of creating and using a binding directory:

  1. Create the binding directory:

    CRTBNDDIR BNDDIR(MYLIB/MYBNDDIR)

  2. Add a module or service program to the binding directory:

    ADDBNDDIRE BNDDIR(MYLIB/MYBNDDIR) OBJ((MYLIB/HELLOMOD *MODULE))

  3. Use the binding directory when creating a program:

    CRTPGM PGM(MYLIB/HELLOPGM) BNDDIR(MYLIB/MYBNDDIR)

Activation Groups

Activation groups manage the runtime environment for ILE programs and service programs. They control resources such as memory and file usage.

Types of Activation Groups

  • Named Activation Groups: Explicitly named by the developer.
  • *NEW: Each call creates a new activation group.
  • *CALLER: Uses the activation group of the calling program.
  • *DFTACTGRP: Uses the default activation group, typically for OPM (Original Program Model) programs.

Specifying an Activation Group

Here is an example of specifying an activation group in an RPG program:

**FREE
Ctl-Opt ActGrp('MYACTGRP');

Dcl-Proc HelloWorld;
  Dcl-S message Char(50);
  message = 'Hello, World!';
  Dsply message;
End-Proc;

In this example, the Ctl-Opt directive specifies that the program should run in the activation group named MYACTGRP.

Conclusion

Understanding ILE concepts and architecture is crucial for developing efficient and modular applications on IBM i systems. By mastering modules, programs, service programs, binding directories, and activation groups, you can create robust and maintainable applications. Practice creating and binding these components to solidify your understanding and enhance your programming skills in RPG.
