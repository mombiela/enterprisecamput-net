Serving Static Content with NGINX
Introduction to HTTP in NGINX ►

Introduction

Serving static content is one of the most common and fundamental uses of NGINX. Static content includes files like HTML, CSS, JavaScript, images, and videos that do not change dynamically. This section will guide you through the basics to advanced techniques for serving static content using NGINX.

Key Concepts

  • Static Content: Files that do not change dynamically and are served as-is.
  • Root Directive: Specifies the root directory from which files are served.
  • Index Directive: Defines the default file to serve when a directory is requested.
  • Location Block: Used to match specific URLs and apply configurations.

Basic Configuration

Setting Up the Root Directory

To serve static content, you need to specify the root directory where your files are stored. This is done using the root directive inside a server block.

server {
    listen 80;
    server_name example.com;

    root /var/www/html;

    location / {
        try_files $uri $uri/ =404;
    }
}
  • listen 80;: NGINX listens on port 80 for HTTP requests.
  • server_name example.com;: Specifies the domain name.
  • root /var/www/html;: Sets the root directory to /var/www/html.
  • location / { ... }: Matches all requests and tries to serve the corresponding file.

Default Index File

You can specify a default file to serve when a directory is requested using the index directive.

server {
    listen 80;
    server_name example.com;

    root /var/www/html;
    index index.html;

    location / {
        try_files $uri $uri/ =404;
    }
}
  • index index.html;: Sets index.html as the default file to serve.

Advanced Configuration

Serving Different Types of Static Content

You can create multiple location blocks to serve different types of static content from different directories.

server {
    listen 80;
    server_name example.com;

    location / {
        root /var/www/html;
        try_files $uri $uri/ =404;
    }

    location /images/ {
        root /var/www/media;
    }

    location /css/ {
        root /var/www/styles;
    }
}
  • location /images/ { root /var/www/media; }: Serves images from /var/www/media.
  • location /css/ { root /var/www/styles; }: Serves CSS files from /var/www/styles.

Caching Static Content

To improve performance, you can enable caching for static content.

server {
    listen 80;
    server_name example.com;

    root /var/www/html;
    index index.html;

    location / {
        try_files $uri $uri/ =404;
    }

    location ~* \.(jpg|jpeg|png|gif|ico|css|js)$ {
        expires 30d;
        add_header Cache-Control "public, no-transform";
    }
}
  • location ~* \.(jpg|jpeg|png|gif|ico|css|js)$ { ... }: Matches requests for images, CSS, and JavaScript files.
  • expires 30d;: Sets the cache expiration to 30 days.
  • add_header Cache-Control "public, no-transform";: Adds a Cache-Control header to the response.

Compression

You can enable gzip compression to reduce the size of the files being served.

http {
    gzip on;
    gzip_types text/plain text/css application/json application/javascript text/xml application/xml application/xml+rss text/javascript;
    gzip_min_length 256;
}
  • gzip on;: Enables gzip compression.
  • gzip_types ...;: Specifies the MIME types to compress.
  • gzip_min_length 256;: Only compress files larger than 256 bytes.

Conclusion

Serving static content with NGINX is straightforward yet powerful. By understanding and utilizing the root and index directives, configuring location blocks, enabling caching, and compression, you can efficiently serve static files to your users. These techniques not only improve performance but also enhance the user experience by delivering content quickly and reliably.
