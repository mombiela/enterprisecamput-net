Introduction

The Chain of Responsibility pattern is a behavioral design pattern that allows multiple objects to handle a request without explicitly coupling the sender of the request to its receiver. This pattern creates a chain of receiver objects and passes the request along the chain until an object handles it.

Learning Objectives

  • Understand the purpose and structure of the Chain of Responsibility pattern.
  • Identify when and how to apply this pattern in software development.
  • Implement the Chain of Responsibility pattern in a practical example.

Key Concepts

  • Responsibility: Each object in the chain has the opportunity to handle the request.
  • Decoupling: The sender of the request does not need to know which object in the chain will handle the request.
  • Flexibility: It is easy to add or change handlers in the chain without modifying the sender's code.

Pattern Structure

The Chain of Responsibility pattern consists of the following components:

  • Handler: Defines an interface for handling requests.
  • ConcreteHandler: Handles requests it is responsible for and passes the request to the next handler in the chain if it cannot handle it.
  • Client: Initiates the request to one of the handlers in the chain.

Practical Example

Let's implement a simple example in Python where different handlers process a technical support request.

Example Code

class Handler:
    def __init__(self, successor=None):
        self._successor = successor

    def handle(self, request):
        raise NotImplementedError("Must implement the handle method")

class LevelOneSupport(Handler):
    def handle(self, request):
        if request == "simple issue":
            return "LevelOneSupport: Resolved simple issue"
        elif self._successor:
            return self._successor.handle(request)

class LevelTwoSupport(Handler):
    def handle(self, request):
        if request == "complex issue":
            return "LevelTwoSupport: Resolved complex issue"
        elif self._successor:
            return self._successor.handle(request)

class LevelThreeSupport(Handler):
    def handle(self, request):
        if request == "critical issue":
            return "LevelThreeSupport: Resolved critical issue"
        elif self._successor:
            return self._successor.handle(request)

# Setting up the chain of responsibility
level_three = LevelThreeSupport()
level_two = LevelTwoSupport(level_three)
level_one = LevelOneSupport(level_two)

# Client makes a request
requests = ["simple issue", "complex issue", "critical issue", "unknown issue"]

for req in requests:
    result = level_one.handle(req)
    print(f"Request: {req} -> {result}")

Code Explanation

  • Handler: Base class that defines the handle interface.
  • LevelOneSupport, LevelTwoSupport, LevelThreeSupport: Concrete handlers that implement the logic to handle different types of requests.
  • Chain setup: A chain of handlers is created where LevelOneSupport passes the request to LevelTwoSupport if it cannot handle it, and so on.
  • Client: Makes several requests and the chain of handlers processes each one.

Conclusion

The Chain of Responsibility pattern is useful for situations where multiple objects can handle a request and the sender does not need to know which object will handle it. This pattern promotes decoupling and flexibility in request handling. When implementing this pattern, it is important to carefully design the chain of handlers to ensure that all requests are processed appropriately.
