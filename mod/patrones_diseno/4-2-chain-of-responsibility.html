Introduction

The Chain of Responsibility pattern is a behavioral design pattern that allows an object to pass a request along a chain of potential handlers until the request is handled. This pattern decouples the sender of a request from its receiver by giving multiple objects a chance to handle the request.

Key Concepts

  • Handler: An interface or abstract class that defines a method for handling requests and a reference to the next handler in the chain.
  • Concrete Handler: A class that implements the handler interface and processes requests it is responsible for. If it can't handle the request, it passes it to the next handler in the chain.
  • Client: The object that initiates the request.

Structure

The Chain of Responsibility pattern typically involves the following components:

  1. Handler Interface/Abstract Class: Defines the method for handling requests and a reference to the next handler.
  2. Concrete Handlers: Implement the handler interface and handle the requests they are responsible for.
  3. Client: Initiates the request and sets up the chain of handlers.

Example

Let's consider a scenario where we have different levels of support: Low, Medium, and High. Each level of support can handle certain types of requests, and if a request is beyond its capability, it passes the request to the next level.

Code Example

class Handler:
    def __init__(self, successor=None):
        self._successor = successor

    def handle_request(self, request):
        raise NotImplementedError("This method should be overridden by subclasses")


class LowLevelSupport(Handler):
    def handle_request(self, request):
        if request == "low":
            print("LowLevelSupport: Handling low-level request")
        elif self._successor:
            self._successor.handle_request(request)


class MediumLevelSupport(Handler):
    def handle_request(self, request):
        if request == "medium":
            print("MediumLevelSupport: Handling medium-level request")
        elif self._successor:
            self._successor.handle_request(request)


class HighLevelSupport(Handler):
    def handle_request(self, request):
        if request == "high":
            print("HighLevelSupport: Handling high-level request")
        elif self._successor:
            self._successor.handle_request(request)


# Setting up the chain of responsibility
high_level = HighLevelSupport()
medium_level = MediumLevelSupport(high_level)
low_level = LowLevelSupport(medium_level)

# Client making requests
requests = ["low", "medium", "high", "unknown"]

for req in requests:
    print(f"Client: Making request '{req}'")
    low_level.handle_request(req)
    print()

Explanation

  1. Handler Class: Defines the interface for handling requests and holds a reference to the next handler.
  2. LowLevelSupport, MediumLevelSupport, HighLevelSupport: Concrete handlers that handle specific types of requests. If they can't handle a request, they pass it to the next handler in the chain.
  3. Client: Sets up the chain of handlers and makes requests.

Output

Client: Making request 'low'
LowLevelSupport: Handling low-level request

Client: Making request 'medium'
MediumLevelSupport: Handling medium-level request

Client: Making request 'high'
HighLevelSupport: Handling high-level request

Client: Making request 'unknown'

Advantages and Disadvantages

Advantages

  • Decoupling: The sender of a request is decoupled from the receiver.
  • Flexibility: The chain can be modified dynamically by changing the order of handlers or adding/removing handlers.
  • Responsibility Sharing: Multiple objects can handle the request, promoting a more distributed approach.

Disadvantages

  • Uncertain Handling: There is no guarantee that a request will be handled if the chain is not properly configured.
  • Debugging Complexity: It can be harder to debug and trace the flow of requests through the chain.

Conclusion

The Chain of Responsibility pattern is a powerful tool for creating flexible and decoupled systems where multiple objects can handle requests. By understanding and implementing this pattern, developers can create more maintainable and scalable software solutions.
