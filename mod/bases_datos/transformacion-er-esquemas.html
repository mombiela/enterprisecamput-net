Introduction

In this section, we will learn how to transform an Entity-Relationship (ER) Diagram into a relational schema. This process is fundamental for designing relational databases from conceptual models, ensuring that the database structure is efficient and coherent.

Key Concepts

  • ER Diagram: Graphical representation of entities and their relationships.
  • Relational Schema: Set of tables that represent the entities and relationships of an ER diagram in a relational database.
  • Entity: Real-world object that is distinguishable from other objects.
  • Attribute: Property or characteristic of an entity.
  • Primary Key: Attribute or set of attributes that uniquely identify an entity.
  • Foreign Key: Attribute in a table that refers to the primary key of another table.

Steps for Transformation

1. Transformation of Simple Entities

Each simple entity in the ER diagram is converted into a table in the relational schema.

  • Example:

    • Entity Student with attributes ID, Name, Age.

    • Resulting table:

      CREATE TABLE Student (
    ID INT PRIMARY KEY,
    Name VARCHAR(100),
    Age INT
);

2. Transformation of Relationships

Relationships between entities are transformed into additional tables or incorporated into existing tables using foreign keys.

One-to-One Relationships (1:1)

  • Example:

    • Entities Person and Passport with a 1:1 relationship.

    • Resulting table:

      CREATE TABLE Person (
    PersonID INT PRIMARY KEY,
    Name VARCHAR(100)
);

CREATE TABLE Passport (
    PassportID INT PRIMARY KEY,
    PersonID INT,
    ExpirationDate DATE,
    FOREIGN KEY (PersonID) REFERENCES Person(PersonID)
);

One-to-Many Relationships (1:N)

  • Example:

    • Entities Professor and Course with a 1:N relationship.

    • Resulting table:

      CREATE TABLE Professor (
    ProfessorID INT PRIMARY KEY,
    Name VARCHAR(100)
);

CREATE TABLE Course (
    CourseID INT PRIMARY KEY,
    Name VARCHAR(100),
    ProfessorID INT,
    FOREIGN KEY (ProfessorID) REFERENCES Professor(ProfessorID)
);

Many-to-Many Relationships (M:N)

  • Example:

    • Entities Student and Course with an M:N relationship.

    • Resulting table:

      CREATE TABLE Student (
    StudentID INT PRIMARY KEY,
    Name VARCHAR(100)
);

CREATE TABLE Course (
    CourseID INT PRIMARY KEY,
    Name VARCHAR(100)
);

CREATE TABLE StudentCourse (
    StudentID INT,
    CourseID INT,
    PRIMARY KEY (StudentID, CourseID),
    FOREIGN KEY (StudentID) REFERENCES Student(StudentID),
    FOREIGN KEY (CourseID) REFERENCES Course(CourseID)
);

3. Transformation of Composite Attributes

Composite attributes are decomposed into simple attributes.

  • Example:

    • Composite attribute Address with sub-attributes Street, City, Postal Code.

    • Resulting table:

      CREATE TABLE Client (
    ClientID INT PRIMARY KEY,
    Name VARCHAR(100),
    Street VARCHAR(100),
    City VARCHAR(100),
    PostalCode VARCHAR(10)
);

4. Transformation of Multivalued Attributes

Multivalued attributes are transformed into separate tables.

  • Example:

    • Entity Employee with multivalued attribute Phones.

    • Resulting table:

      CREATE TABLE Employee (
    EmployeeID INT PRIMARY KEY,
    Name VARCHAR(100)
);

CREATE TABLE Phone (
    EmployeeID INT,
    PhoneNumber VARCHAR(15),
    PRIMARY KEY (EmployeeID, PhoneNumber),
    FOREIGN KEY (EmployeeID) REFERENCES Employee(EmployeeID)
);

Practical Exercise

Transform the following ER diagram into a relational schema:

  • Entities: Author (ID, Name), Book (ISBN, Title, Year), Publisher (ID, Name).
  • Relationships:
    • Author writes Book (M:N).
    • Book is published by Publisher (N:1).

Conclusion

The transformation of ER diagrams to relational schemas is a crucial step in database design. This process ensures that the database structure is logical and efficient, facilitating the implementation and maintenance of the database. Practicing these steps with different ER diagrams will help consolidate these concepts and skills.
