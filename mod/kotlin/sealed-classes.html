Sealed Classes in Kotlin
Generics ►

Sealed classes are a powerful feature in Kotlin that allow you to represent restricted class hierarchies. They are particularly useful when you have a fixed set of types that you want to work with, making your code more robust and easier to understand.

What are Sealed Classes?

  • Definition: Sealed classes are classes that restrict the hierarchy to a limited set of subclasses. This means that all possible subclasses are known at compile time.
  • Purpose: They are used to model restricted class hierarchies, where a value can have one of the types from a limited set.

Declaring Sealed Classes

To declare a sealed class, you use the sealed keyword before the class declaration.

sealed class Shape {
    data class Circle(val radius: Double) : Shape()
    data class Rectangle(val height: Double, val width: Double) : Shape()
    object NotAShape : Shape()
}

Key Points

  • Cannot be instantiated: You cannot create an instance of a sealed class directly.
  • Subclasses must be nested: All subclasses of a sealed class must be defined in the same file.

Using Sealed Classes

Sealed classes are typically used with when expressions to handle different types in a type-safe manner.

fun describeShape(shape: Shape): String {
    return when (shape) {
        is Shape.Circle -> "A circle with radius ${shape.radius}"
        is Shape.Rectangle -> "A rectangle with height ${shape.height} and width ${shape.width}"
        Shape.NotAShape -> "Not a shape"
    }
}

Example Explained

  • when expression: The when expression is exhaustive, meaning it covers all possible subclasses of the sealed class.
  • Type safety: The compiler ensures that all cases are handled, making your code safer and less prone to errors.

Advantages of Sealed Classes

  • Exhaustiveness: The compiler checks that all possible cases are handled, reducing runtime errors.
  • Readability: Code is easier to read and understand because the hierarchy is well-defined and limited.
  • Maintainability: Adding new types is straightforward and less error-prone.

Comparison with Enum Classes

Feature Sealed Classes Enum Classes
Hierarchy Can have multiple subclasses Single type with fixed constants
Data Can hold different data types Limited to predefined constants
Extensibility More flexible, can add new subclasses Less flexible, fixed set of constants

Practical Exercise

Create a sealed class hierarchy to represent different types of responses from a network request.

  1. Define a sealed class NetworkResponse with the following subclasses:

    • Success with a data property of type String.
    • Error with a message property of type String.
    • Loading with no properties.

  2. Write a function handleResponse that takes a NetworkResponse and returns a message based on the type of response.

sealed class NetworkResponse {
    data class Success(val data: String) : NetworkResponse()
    data class Error(val message: String) : NetworkResponse()
    object Loading : NetworkResponse()
}

fun handleResponse(response: NetworkResponse): String {
    return when (response) {
        is NetworkResponse.Success -> "Data received: ${response.data}"
        is NetworkResponse.Error -> "Error occurred: ${response.message}"
        NetworkResponse.Loading -> "Loading..."
    }
}

Conclusion

Sealed classes in Kotlin provide a robust way to define restricted class hierarchies, ensuring type safety and making your code more readable and maintainable. They are particularly useful when you need to handle a fixed set of types, and they work seamlessly with Kotlin's when expressions to provide exhaustive checks at compile time. By understanding and utilizing sealed classes, you can write more reliable and maintainable Kotlin code.
