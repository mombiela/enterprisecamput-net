Introduction

Schema design is a fundamental part of creating efficient and scalable databases. A well-designed schema facilitates data management, improves query performance, and ensures data integrity. In this section, we will explore the key principles of schema design, providing examples and exercises to reinforce the concepts.

Key Concepts

  • Normalization: Process of structuring a database to reduce redundancy and improve data integrity.
  • Denormalization: Process of combining tables to improve query performance at the cost of redundancy.
  • Referential Integrity: Ensures that relationships between tables remain consistent.
  • Cardinality: Defines the nature of relationships between entities (one-to-one, one-to-many, many-to-many).
  • Primary and Foreign Keys: Unique identifiers for records and references between tables.

Normalization

Normalization is the process of organizing data in a database to reduce redundancy and improve data integrity. There are several normal forms (NF) applied in this process:

  • First Normal Form (1NF): Eliminate repeating groups and ensure each field contains only one atomic value.
  • Second Normal Form (2NF): Ensure all non-key attributes depend entirely on the primary key.
  • Third Normal Form (3NF): Eliminate transitive dependencies, ensuring non-key attributes depend only on the primary key.

Example of Normalization

Suppose we have an employee table with the following structure:

| EmployeeID | Name | Department | DepartmentHead | |------------|-------|------------|----------------| | 1 | Alice | HR | Bob | | 2 | John | IT | Carol | | 3 | Mike | HR | Bob |

To normalize this table to 2NF, we can split it into two tables:

Employee Table:

| EmployeeID | Name | DepartmentID | |------------|-------|--------------| | 1 | Alice | 1 | | 2 | John | 2 | | 3 | Mike | 1 |

Department Table:

| DepartmentID | Department | DepartmentHead | |--------------|------------|----------------| | 1 | HR | Bob | | 2 | IT | Carol |

Denormalization

Denormalization is the process of combining tables to improve query performance at the cost of introducing redundancy. It is useful in scenarios where read queries are more frequent than write queries.

Example of Denormalization

If frequent queries require information from both employees and departments, we could denormalize the previous tables into one:

| EmployeeID | Name | Department | DepartmentHead | |------------|-------|------------|----------------| | 1 | Alice | HR | Bob | | 2 | John | IT | Carol | | 3 | Mike | HR | Bob |

Referential Integrity

Referential integrity ensures that relationships between tables remain consistent. This is achieved through the use of foreign keys.

Example of Referential Integrity

In the employee table, DepartmentID is a foreign key that references DepartmentID in the department table. This ensures that each employee is associated with a valid department.

CREATE TABLE Departments (
    DepartmentID INT PRIMARY KEY,
    DepartmentName VARCHAR(50),
    DepartmentHead VARCHAR(50)
);

CREATE TABLE Employees (
    EmployeeID INT PRIMARY KEY,
    Name VARCHAR(50),
    DepartmentID INT,
    FOREIGN KEY (DepartmentID) REFERENCES Departments(DepartmentID)
);

Cardinality

Cardinality defines the nature of relationships between entities. Relationships can be:

  • One to One (1:1): One record in one table is related to a single record in another table.
  • One to Many (1:N): One record in one table is related to multiple records in another table.
  • Many to Many (M:N): Multiple records in one table are related to multiple records in another table.

Example of Cardinality

One to Many (1:N): A department has many employees.

CREATE TABLE Departments (
    DepartmentID INT PRIMARY KEY,
    DepartmentName VARCHAR(50)
);

CREATE TABLE Employees (
    EmployeeID INT PRIMARY KEY,
    Name VARCHAR(50),
    DepartmentID INT,
    FOREIGN KEY (DepartmentID) REFERENCES Departments(DepartmentID)
);

Primary and Foreign Keys

  • Primary Key: A unique identifier for each record in a table.
  • Foreign Key: A field in one table that is a primary key in another table, used to establish and enforce a link between the data in the two tables.

Example of Primary and Foreign Keys

CREATE TABLE Departments (
    DepartmentID INT PRIMARY KEY,
    DepartmentName VARCHAR(50)
);

CREATE TABLE Employees (
    EmployeeID INT PRIMARY KEY,
    Name VARCHAR(50),
    DepartmentID INT,
    FOREIGN KEY (DepartmentID) REFERENCES Departments(DepartmentID)
);

Conclusion

Schema design is crucial for the efficiency and integrity of a database. Normalization and denormalization are complementary techniques that should be applied according to the specific needs of the system. Referential integrity and the correct definition of cardinality ensure that data remains consistent and accurate. Understanding and applying these principles is essential for any IT professional working with databases.
